## 冒烟测试

### SMK-01: 环境启动

- **目标：** 确认应用可正常启动和访问
- **前置条件：** MySQL 8 运行中，已执行 seed
- **步骤：**
  1. `npm run build`（无报错）
  2. `npm run dev`
  3. 浏览器打开 http://localhost:3000
- **预期：** 显示首页，含分类导航和推荐商品
- **状态：** 未测
- **备注：** 任何后续测试的前置条件

### SMK-02: 注册新用户

- **目标：** 验证注册流程完整可用
- **前置条件：** 未登录
- **步骤：**
  1. 点击"登录" → "立即注册"
  2. 填写昵称、邮箱使用 `smoke+<当前时间戳>@example.com`（如 `smoke+1719000000@example.com`），密码任意
  3. 点击"注册"
- **预期：** 跳转首页，导航栏显示昵称和退出按钮
- **状态：** 未测
- **备注：** 使用时间戳邮箱避免重复注册导致失败；注册即登录

### SMK-12: 下单

- **目标：** 完整下单流程：结账 → 订单创建 → 库存扣减 → 购物车清空
- **前置条件：** 购物车有商品，记录某商品库存为 S
- **步骤：**
  1. 购物车页点击"去结算"
  2. 填写收货人姓名、电话、地址
  3. 点击"提交订单"
  4. 查看订单详情
  5. 回到商品详情页查看库存
- **预期：** 订单创建成功，显示订单号和商品快照；购物车已清空；库存变为 S - 购买数量
- **状态：** 未测
- **备注：** 这是最核心的冒烟用例，验证事务完整性

### SMK-13: 模拟支付

- **目标：** PENDING 订单可以支付
- **前置条件：** 有一个 PENDING 订单
- **步骤：**
  1. 进入订单详情页
  2. 点击"模拟支付"
- **预期：** 订单状态变为"已支付"（蓝色徽章），支付按钮消失
- **状态：** 未测
- **备注：**

### SMK-14: 取消 PENDING 订单并恢复库存

- **目标：** 取消订单后状态变更 + 库存恢复
- **前置条件：** **重新创建一个新的 PENDING 订单**（SMK-13 已支付的订单不能取消，必须另建一个），记录其中商品当前库存
- **步骤：**
  1. 进入该 PENDING 订单详情
  2. 点击"取消订单"并确认
  3. 回到商品详情页查看库存
- **预期：** 状态变为"已取消"；库存恢复至下单前数量
- **状态：** 未测
- **备注：** 验证事务原子性（状态变更 + 库存恢复在同一事务内）。注意：SMK-13 支付后将订单变成 PAID，PAID 不允许取消，所以 SMK-14 必须另外创建一个新的 PENDING 订单

### SMK-15: 库存不足时下单失败

- **目标：** 防超卖机制生效
- **前置条件：** 以 ADMIN 登录，将某个上架商品库存改为 1；以 USER 登录，将该商品加入购物车（数量 2）
- **步骤：**
  1. 访问 /checkout，填写收货信息
  2. 点击"提交订单"
- **预期：** 下单失败，提示"库存不足"
- **状态：** 未测
- **备注：** 验证条件更新 `stock >= quantity` 校验生效。双用户并发防超卖属于回归测试范围，见 [08-test-cases.md](08-test-cases.md) TC-ORDER 并发用例

### 覆盖的核心流程

```
注册 → 登录 → 浏览商品 → 加入购物车 → 下单(扣库存) → 支付 → 取消(恢复库存)
                                              ↑
                     ADMIN 后台管理商品/订单状态变更
```

### 执行时间估算

约 20~30 分钟（含准备和切换账号）。
